## VERIFIED: what `phasepoly fasttodd` actually does (2026-06-17, read end-to-end)

> This section is the authoritative, code-grounded account. It supersedes the
> older sections below where they conflict — in particular the framing that the
> path is a faithful, *efficient* Vandaele FastTODD. The construction is
> FastTODD-shaped, but the **fast solver is not implemented**: the path's cost
> is TODD-class, not FastTODD-class.

**Dispatch chain (verified).** `tableau optimize phasepoly fasttodd`
→ `src/cmd/tableau_cmd.cpp:267-268` (`is_prefix_of(..., "fasttodd")`
→ `FastToddPhasePolynomialOptimizationStrategy`) → `fastTodd.cpp:855`.

**Control flow (verified by reading the bodies, not the names).**

- `FastToddPhasePolynomialOptimizationStrategy::optimize` (`:855`): reject
  non-4th-root phases → `properize` → snapshot `MultiLinearPolynomial` → loop
  `ret_polynomial = fasttodd_once(...)` until the term count stops shrinking
  (`:877-885`) → `extract_clifford_operators` (`:889`).
- `fasttodd_once` (`:753`): (1) **re-runs the full TOHPE fixpoint** —
  `tohpe_once_iteration` in a loop until the count stops dropping (`:759-765`);
  (2) then **exactly one** `fast_todd_iteration` move (`:770`).
- `fast_todd_iteration` (`:690`): `build_l` (pairwise AND over qubit pairs,
  `:698`); for each candidate `z` from `generate_candidate_z` (every term column
  + every pairwise column XOR, ~`O(m²)` candidates, `:705`): build `X`/`v`,
  `constraint = hstack(L, X, v)` (`:718`), `nullspace = get_nullspace_transposed`
  (`:719`, full Gaussian elimination), then `iterate_nullspace_combinations`
  (`:725`) and take the **first** `(z,y)` that reduces the term count
  (`:743-746`).

**Why this is *not* FastTODD's algorithm (the central finding).**

1. **Exhaustive nullspace enumeration instead of the fast solve.**
   `iterate_nullspace_combinations` (`:643`) loops `mask` over
   `1 .. 2^num_basis − 1` (`:652`) and XORs basis vectors — i.e. it materializes
   **every** element of the per-`z` solution space. Vandaele's whole point
   (Theorem 6, Eq 86-90) is to read the solution set `N^(z)` directly off the
   reduced column-echelon form in polynomial time; **that fast solve is absent.**
   This is what gives FastTODD `O(n²m³)`; without it the per-`z` work is
   exponential in the nullspace dimension. This is the single biggest deviation.
2. **Per-`z` full nullspace recompute.** A fresh Gaussian elimination runs for
   every candidate `z` (`:719`), inside the `O(m²)`-candidate loop.
3. **Redundant TOHPE fixpoints.** `fasttodd_once` runs TOHPE *to fixpoint* before
   **each** single FastTODD move, and `optimize` loops `fasttodd_once`; so the
   number of full TOHPE fixpoints scales with the number of FastTODD moves.
4. **First-improving, not argmax.** `:743-746` returns the first move with
   `score>0`; Vandaele Alg 3 / Eq 85 takes the argmax. Correctness-preserving,
   but means more outer iterations.
5. **No signature-tensor guard.** `Tohpe…::optimize` verifies
   `has_same_signature_tensor` post-optimization (`:838`); the FastTODD driver
   (`:855-897`) has **no** equivalent check — its only safety net is
   `extract_clifford_operators` returning a value.

**Verdict.** The `fasttodd` path implements FastTODD's *algebraic construction*
(`build_l`/`build_x`/`build_v` + `apply_rank1_update` match Thm 6 / Eq 63-67, 84)
and does produce FastTODD-quality T-counts on the small/mid circuits (output
tracks the Vandaele FastTODD column). But the **performance-critical fast solver
is replaced by `2^num_basis` brute-force enumeration**, so its runtime profile is
TODD-class, not FastTODD-class. Empirical corroboration: Vandaele reports
`hwb11` (44.5k T) in **28 s**, whereas this path ran `gf2^16` for **>1 h** in the
current benchmark. So "FastTODD" is accurate about the *quality target* and the
*matrix construction*, but misleading about the *algorithm/complexity*: the
defining "fast" component (Eq 86-90) is not there.

**Cross-check of the older analysis below.** Its function→line map and the issue
list (#1 first-move, #2 `2^num_basis`, #3 missing guard, #4 slow/fast TOHPE
divergence, #5 debug scaffolding) all check out against the current source. Its
error is only in the *characterization* — presenting the path as an efficient
FastTODD rather than a FastTODD construction with the fast solve missing.

## Paper anchors (verified)

- **TODD** — Heyfron & Campbell, "Third Order Duplicate and Destroy". Signature
  tensor framework: extract a symmetric order-3 tensor `S` from the phase gate,
  find a low-rank decomposition `S_{α,β,γ}=⊕_j A_{α,j}A_{β,j}A_{γ,j}`, resynthesize
  (Fig. 1).
- **Framework** — Vandaele §2: weighted polynomial **Eq. 8**, signature tensor
  **Eq. 9**, phase polynomial / parity table `P` **Eq. 10**; T-count
  minimization = third-order symmetric tensor rank (3-STR) problem,
  **Problem 1 / Eq. 16–17**.
- **TOHPE** = "Third order homogeneous polynomials elimination" = **Algorithm 2**
  (Vandaele p.12), built on **Theorem 1** (Eq. 18–21). Candidate set
  `Z = {P_{:,i}⊕P_{:,j}} ∪ {P_{:,i}}` (**Eq. 33**); `L_{αβ}=P_α∧P_β` (**Eq. 34**);
  move selection by **argmax** of the objective **Eq. 35** (diagonal case
  **Eq. 36**). Complexity `O(n²m³)`.
- **Base TODD** = Vandaele §3.2, **Theorem 4** (Eq. 37–40), via a χ-matrix;
  complexity `O(n³m⁵)` reduced to `O(n⁴m³)`.
- **FastTODD** = **Algorithm 3** (Vandaele p.18), built on **Theorem 6**
  (Eq. 63–67): `L_{αβ}=P_α∧P_β` (**Eq. 63**), `X_{αβ,γ}=z_α δ_{βγ}⊕z_β δ_{αγ}`
  (**Eq. 64**), `v_{αβ}=z_α∧z_β` (**Eq. 65**), homogeneous system
  `Ly ⊕ Xy' ⊕ bv = 0` (**Eq. 67/83**), rank-1 update `P'=P⊕zyᵀ` (append column
  `z` when `|y|` odd) (**Eq. 84**), move selection by **argmax** of **Eq. 85**
  (identical objective to Eq. 35). The per-`z` solution set `N^{(z)}` is computed
  *efficiently* from the reduced column-echelon form of `L` (**Eq. 86–90**),
  giving the `O(n²m³)` complexity — **not** by exhaustive enumeration.

## Notes per construct

- **Two TOHPE paths.** `Tohpe…::optimize` defaults to the incremental path:
  `tohpe_use_iteration()` (env `QSYN_TOHPE_USE_ITERATION`) defaults true, so the
  slow `tohpe_once` runs only when explicitly disabled or under
  `QSYN_TOHPE_TRACE_COMPARE`. `fasttodd_once` always uses the incremental path
  (line 761). The slow path is therefore effectively a reference oracle.
- **Move selection vs. paper.** Both TOHPE paths already match the paper: pick a
  `y` from the `L`-nullspace, then choose `z` by **argmax** of the Eq. 35
  objective (slow: `calculate_score` over `s_matrices`, lines 356–365; fast: the
  incremental `map` argmax with `compare_row_value` tie-break, lines 445–455).
  The difference is which `y` each path takes (slow: first candidate nullspace
  row; fast: `kernel(...)` result) — the source of the divergence in issue #4.
- **FastTODD candidate/solve.** `fast_todd_iteration` matches Theorem 6's
  `L`/`X`/`v` construction and the `Ly⊕Xy'⊕bv=0` system exactly, but the search
  strategy differs from Algorithm 3 (issues #1, #2).

## Known issues / limitations (hand-off targets)

1. **FastTODD takes the first improving move, not the argmax.**
   `fast_todd_iteration` returns the first `(z,y)` with `score>0`
   (`fastTodd.cpp:743-745`), whereas Algorithm 3 / Eq. 85 selects the `(z,y)`
   pair **maximizing** the removed-column count. (TOHPE is *not* affected — both
   TOHPE paths already argmax `z`.)
2. **Exponential nullspace enumeration.** `iterate_nullspace_combinations`
   enumerates all `2^num_basis−1` basis combinations (`fastTodd.cpp:652`), called
   per candidate `z` inside `fast_todd_iteration`. The paper computes `N^{(z)}`
   directly via reduced column-echelon form (Eq. 86–90); this is exactly what
   gives FastTODD its `O(n²m³)` complexity. The current enumeration defeats
   FastTODD's reason to exist and risks blow-up on large nullspaces.
3. **Missing signature-tensor guard in FastTODD.** `Tohpe…::optimize` validates
   `has_same_signature_tensor` after optimization (`fastTodd.cpp:838`); the
   FastTODD driver has no equivalent check (`fastTodd.cpp:877-885`) and relies
   solely on `extract_clifford_operators` returning a value. Add the guard or
   justify why it is unnecessary by construction.
4. **Slow vs. fast TOHPE divergence.** The `QSYN_TOHPE_TRACE_COMPARE` /
   `..._STOP_ON_DIVERGENCE` scaffolding exists to catch cases where
   `tohpe_once` and `tohpe_once_iteration` disagree (`fastTodd.cpp:812-826`).
   Decide on one correct path; make them agree or drop the slow one.
5. **Leftover debug scaffolding.** `QSYN_TOHPE_TRACE*` env reads, `tohpe_trace`,
   `trace_iteration_tohpe`, and the `row_to_bitstring`/`bytes_to_bitstring`/
   `summarize_*` helpers (`fastTodd.cpp:37-119`, call sites through ~558) should
   be removed once #4 is resolved.

## 2×2 benchmark: qsyn vs Vandaele reference impl (2026-06-18)

Head-to-head against Vandaele's official Rust impl
(`github.com/VivienVandaele/quantum-circuit-optimization`, cloned to
`/home/liao/quantum-circuit-optimization`). Both tools are **single-threaded**
(no rayon/threads in either). Matrix = {qsyn, Vandaele} × {pyzx inputs, raw
inputs}; qsyn pipeline `tmerge→hopt→phasepoly fasttodd`, Vandaele default
`FastTMerge→InternalHOpt→FastTODD`. Harness: `bench2x2/run.sh` (persistent,
resumable, `timeout -k 10`, `ulimit -v 12 GB`→`MLE`). Outputs: `bench2x2/all.csv`
(154 rows) + `bench2x2/table.md`. Caps: 1800s/circuit, 4-way parallel.

**Three regimes:**
1. **Small/mid circuits (no internal H):** both finish instantly; Vandaele gets
   slightly *lower* T on raw inputs (e.g. adder_8 119 vs qsyn 170).
2. **`gf2^*` family (qsyn weakness):** both correct, qsyn ~70–100× slower
   (gf2^9: qsyn 197s vs Vandaele 3s); qsyn **TLEs gf2^16**, Vandaele 140s. This
   is qsyn's missing fast solver (the `2^num_basis` enumeration, issue #2 above).
3. **Hadamard-dense circuits (Vandaele weakness):** roles flip — qsyn finishes
   in seconds (shallow pass), Vandaele **TLE/MLE**.

Failure-mode split: every qsyn failure is **TLE** (it stays memory-light, ~5.5 GB
even on DEFAULT); every Vandaele failure on big circuits is **MLE**. `DEFAULT`
(640q) defeats both — qsyn TLE, Vandaele MLE (measured peak **30.6 GB RSS** under
a 40 GB cap, still climbing → needs >40 GB; aborted SIGABRT, never completed).
qsyn's regime-3 "wins" are shallow (hwb8 3517→3502) and equivalence was **not**
checked in this harness — caveat before citing.

**Input-format limits (observed, not assumed):** Vandaele rejects CZ gates
(`Operator not implemented: Z`) → the real pyzx circuits (41/47 contain CZ) are
`N/A` for Vandaele. Note `DEFAULT`/`hwb10`/`hwb11` use a `vandaele2024/*.qc` file
for *both* columns (user copied Vandaele's raw circuit; no pyzx variant exists),
so for those 3 pyzx≡raw byte-identical and the matching cells are expected.

### Root cause of the Vandaele TLE/MLE — internal-Hadamard gadgetization blowup

The default pipeline gadgetizes **every** internal Hadamard (+1 ancilla qubit
each). FastTODD's `L` matrix is over qubit-pairs (~n²/2 rows), so cost explodes
with qubit count. Internal-H count after `InternalHOpt` predicts the outcome
exactly:

| circuit | internal-H | gadgetized n | outcome |
|---|--:|--:|---|
| gf2^9 / gf2^16 | 0 | 27 / 48 | fast / 140s |
| mod_adder_1024 | 304 | ~330 | TLE |
| ham15-high | 331 | ~350 | TLE |
| hwb8 | 1103 | ~1115 | TLE |
| hwb10 | 5191 | ~5200 | MLE |
| hwb11 | 14441 | ~14460 | MLE |

`InternalHOpt` barely dents hwb (hwb11: 14454→14441) — hwb is intrinsically
Hadamard-dense. You cannot gadgetize 14k Hadamards and run FastTODD in 28s, so
**the paper's seconds-scale hwb/ham15 numbers cannot come from this
gadgetize-all path.**

**SIMD note (faithfulness):** the original sweep accidentally ran Vandaele on the
**scalar** path — a plain `cargo build -r` leaves avx2 disabled (default x86-64 =
sse2 only; `cfg(target_feature="avx2")` false). Rebuilt with
`RUSTFLAGS="-C target-cpu=native"` (enables avx2; CPU also has avx512f, but the
code's SIMD path is avx2/`__m256i` — same as the paper) and **re-ran the whole
Vandaele column**. Result: **every outcome identical** — ham15-high/hwb8/mod_adder
raw TLE'd again at ~1948–1951s (scalar ~1928–1929s), all MLE/N/A unchanged,
completing-circuit times within contention noise. So AVX2 is conclusively **not**
the cause; the gadgetize-all blowup is the wall regardless of SIMD. The published
table is AVX2-faithful.

### 52 GB re-run — separating time-bound vs memory-bound (AVX2 build)

Re-ran the 3 MLE circuits with `ulimit -v 52 GB`, 1800s cap, `/usr/bin/time -v`:
- **hwb10 → TLE(1800s)**, peak ~27–41 GB (under cap). So hwb10 is **time-bound**;
  its 12 GB MLE in the main table was a **cap artifact**, not a real memory wall.
- **hwb11 → MLE(>52 GB)** — genuinely memory-bound (~14k gadgetized qubits;
  `L` matrix alone exceeds 52 GB).
- **default → MLE(>52 GB)** — genuinely memory-bound (matches the earlier 40 GB
  `/usr/bin/time` run: 30.6 GB RSS and still climbing when killed).

Net: of the 6 MLE rows, only hwb10 (2 rows, identical file) is a cap artifact;
hwb11 and default truly need >52 GB. All are intractable for the gadgetize-all
path either way (time or memory). Table footnote to add: `hwb10 = MLE@12GB but
TLE@52GB (time-bound)`.

### Version provenance (why it's not "the paper version")

Repo HEAD `231e6fe` (2026-06-15). Key dates:
- **FastTODD `src/t_opt.rs` written 2024-07-12 (`8857cbb`)** — matches the paper
  (arXiv 2407.08695). Diff `8857cbb`→HEAD for `t_opt.rs` and `circuit.rs`
  (gadgetization) is **only `unsafe fn`→`fn`** — bodies byte-identical. So the
  **algorithm is unchanged** since the paper.
- Substantive post-paper changes: (1) `bit_vector.rs` — `2dc24c9` "Make avx2
  optional and remove avx512" made the avx2 path `cfg(target_feature="avx2")`
  and removed a separate avx512 variant; **speed only, same results**. NOTE the
  paper-era code (`8857cbb`) is itself **AVX2** (`__m256i`,
  `target_feature="avx2"`), *not* AVX512 — so our HEAD build with
  `RUSTFLAGS="-C target-cpu=native"` (avx2 enabled) is **SIMD-faithful to the
  paper**; no avx512 rebuild needed. (2) **FastTMerge bug fix** (`231e6fe`,
  t_merge/pauli_product; preprocessing, not FastTODD/gadgetization).
- **The `hwb*`/`ham15-*` input files were added 2025-12-10 (`8f9fd1e`), ~17
  months after the paper** — they did NOT exist in `circuits/inputs/` at paper
  time (paper-era inputs: adder/barenco/csla/gf2/mod/qcla/shor/tof/vbe only).

**Conclusion:** the hwb/ham15 TLE/MLE is entirely an **input-provenance artifact**
— the paper-faithful algorithm fed late-added, un-preprocessed Hadamard-dense
circuits via gadgetize-all. Not a build error, not an algorithm-version confound.
Checking out `8857cbb` would behave identically on these inputs (and is also
AVX2 — no SIMD difference from our build), so it was not done.

## Phase 2 — Correctness & fixes (2026-06-18)

Implemented the real FastTODD fast solver and the related reliability fixes.
All changes in `src/tableau/optimize/fastTodd.cpp` unless noted.

### What changed

1. **Ported the FastTODD fast solver (bugs #1 + #2).** Replaced
   `fast_todd_iteration` (per-`z` full Gaussian nullspace +
   `iterate_nullspace_combinations` over `2^num_basis` + first-improving move)
   with `fast_todd_move`, a faithful port of Vandaele's `fast_todd`
   (`quantum-circuit-optimization/src/t_opt.rs:228-373`): the reduced
   column-echelon form of `L` is computed **once** via `kernel`; for each
   candidate `z = P_i ⊕ P_j` the `X^(z)`/`v^(z)` columns are reduced against that
   echelon form (Thm 6, Eq. 86-92) and the solution `y` is read straight off the
   augmented matrix (Eq. 90); the **argmax** of the Eq. 85 objective is applied
   (Alg. 3 line 11). Uses qsyn's existing `kernel` / `build_l_transpose_row_from_term`
   (get_l_matrix layout: `n` linear + strict `C(n,2)` quadratic positions).
   `from_boolean_matrix` already cancels duplicate/zero columns, so no separate
   `proper` is needed and the Eq. 85 score matches the realized reduction.
   Deleted the now-dead helpers `build_pair_indices`/`build_l`/`build_x`/`build_v`/
   `generate_candidate_z`/`iterate_nullspace_combinations`/`apply_rank1_update`.
2. **Signature-tensor guard in FastTODD (bug #3).** `FastTodd…::optimize` now runs
   the same `has_same_signature_tensor` post-check as `Tohpe…::optimize` and bails
   to the input clifford/poly on mismatch.
3. **Slow/fast TOHPE + scaffolding (bugs #4, #5).** Kept `tohpe_once` (env
   `QSYN_TOHPE_USE_ITERATION=0`) as a reference path; removed all
   `QSYN_TOHPE_TRACE*` env reads, `tohpe_trace`/`trace_iteration_tohpe` and the
   `row_to_bitstring`/`bytes_to_bitstring`/`summarize_*` helpers, and the
   compare/stop-on-divergence block in `Tohpe…::optimize`. Also removed a stray
   **unguarded** `augmented_matrix[i].print_row()` debug dump inside `kernel`
   (`:140`) that was polluting `phasepoly tohpe`/`fasttodd` output.

   **Divergence finding (re: bug #4):** slow `tohpe_once` and fast
   `tohpe_once_iteration` do **not** agree, and this is not a bug — both preserve
   the signature tensor but pick a different nullspace `y` (slow = first candidate
   row; fast = `kernel` result), so they reach different greedy optima. The **fast
   path is strictly better** on the gf2 family (final T, full pipeline):
   `gf2^4 51/58, gf2^5 79/83, gf2^6 121/124, gf2^7 157/172` (fast/slow). The fast
   path is the default and is what the golden tests pin; `tohpe_once` is retained
   only as a weaker env-gated reference.

*does* complete — **T=791, equiv pass** — but only at **1861 s**, i.e. still over
the project's standard **1800 s** cap, so it remains a **TLE** by the slides'
methodology (it finishes only with an extended 3000 s cap). The genuinely-fixed
circuits are the ones that now finish well under 1800 s (gf2^7 17→2 s, gf2^9
194→15 s, gf2^8, gf2^10). gf2^16 is ~11× slower than Vandaele (raw input,
**T=797 in 168 s**, measured fresh on the AVX2 build): the algorithm is now the
correct `O(n⁴m³)` FastTODD, but
qsyn's `dvlab::BooleanMatrix::Row` is **one byte per bit** (not bit-packed)
whereas Vandaele uses packed `u64`/AVX2 lanes — a large constant factor that
bites at `n = 48` (`L` width `n + C(n,2) = 1176`). (T-counts aren't comparable:
different inputs — qsyn pyzx-pre-optimized vs Vandaele raw, which Vandaele's
`FastTODD`-only rejects on `ccz` so its default pipeline is used.) Closing the
remaining speed gap needs a bit-packed row type / SIMD — out of Phase-2 scope
(data-structure, not algorithm); recorded as future work.
